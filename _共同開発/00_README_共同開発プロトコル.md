# AGO_SYSTEM_MANAGER 共同開発プロトコル

作成: 2026-04-27 日本ネオン株式会社 本部長AI（移植元）
適用: 日本ネオン株式会社 AI ⇄ AGOグループ AI の双方向協業

---

## 0. このプロトコルの位置付け

AGO_SYSTEM_MANAGER は **AGOグループの③受託事業 実案件第1号** であり、菅野社長から日本ネオン株式会社への**初の正式依頼**です。

このため、本プロジェクトは:
- **両社のAIが交互に引き継いで開発できる体制**で進めます
- どちらが触っても**最新状態が共有される仕組み**を持ちます
- 引き継ぎ時の**情報ロス・判断ロスをゼロ**に近づけます

将来の③受託事業の標準フォーマットとしても活用できる設計です。

## 1. 開発体制（両社AI構成）

### 日本ネオン側
- **担当窓口**: 日本ネオン本部長AI（メイン会話）
- **実装担当**: system-director / system-engineer / system-designer / system-qa（既存・本部長配下）
- **作業環境**: `C:\Users\owner\Desktop\ネオン物販\AGO_SYSTEM_MANAGER\`

### AGOグループ側
- **担当窓口**: AGO本部長AI（メイン会話）
- **実装担当**: AGO本部長AI 配下に同名の system-* AI を起動（同梱の事業部雛形 dept-*.template から構築）
- **作業環境**: `C:\Users\<AGO菅野ユーザー>\Desktop\AI版AGOグループ\AGO_SYSTEM_MANAGER\`

### 同期方法

**当面: SSD 経由の手動同期**（小野寺社長が AGO 訪問時に持参）
- AGO で作業 → 帰宅後に SSD 経由で日本ネオン側に取り込み
- 日本ネオン側で作業 → 次回 AGO 訪問時に SSD 持参 → AGO PCに反映

#### ⚠️ SSD 同期時の絶対遵守事項（Git履歴破損防止）

- **`.git/` フォルダはコピー対象から必ず除外する**
  - 日本ネオン側 `ネオン物販/AGO_SYSTEM_MANAGER/` には `.git/` が存在（main ブランチ・origin 接続済）
  - AGO 側には `.git/` 無し
  - フォルダ全体を上書きコピーすると Git 履歴が破損 or remote 設定が混入する
- **コピー対象**: `_共同開発/`・コードファイル群（`demo.html` `index.html` `config.js` `api.php` `subscribe.php` `sw.js` `manifest.json` `docs/` 等）・`.claude/agents/`
- **コピー除外**: `.git/`・`node_modules/`（あれば）・OS固有一時ファイル
- **推奨手順**:
  ```
  robocopy <src> <dst> /E /XD .git node_modules
  または
  エクスプローラーで .git/ を選択せずに必要ファイルだけコピー
  ```

**将来: GitHub 経由の同期**（菅野社長 GitHub アカウント取得後）
- 既存リポジトリ `github.com/nihonneon2025/ago-system-demo` を菅野社長アカウントへ移管 or コラボレーター追加
- 両社AIが clone → pull → 編集 → push の流れ
- 移管/追加判断は菅野社長と小野寺社長の合意事項（ロードマップ Phase 4-A）

## 3. 引き継ぎプロトコル（必須遵守）

### 3-A. 作業開始前（pull / 取り込み時）

両社AIは作業開始前に必ず以下を実施：

1. `_共同開発/01_作業ステータス_最新.md` を Read
2. 直近の `_共同開発/引き継ぎログ/YYYY-MM-DD_*.md` を Read（最新3件）
3. 前回担当者の作業内容・申し送り事項・未完了タスクを把握
4. 自分の担当範囲を確認・着手

### 3-B. 作業完了後（push / 引き渡し前）

両社AIは作業完了後に必ず以下を実施：

1. `_共同開発/01_作業ステータス_最新.md` を更新（後述のテンプレ）
2. `_共同開発/引き継ぎログ/YYYY-MM-DD_<from>→<to>_引き継ぎ.md` を新規作成
3. 過去のステータスは `_共同開発/作業ステータス/YYYY-MM-DD_HHmm_status.md` にスナップショット保存
4. SSD 同期（または git push）で相手側に反映

### 3-C. 衝突時（両社が同じファイルを編集してしまった場合）

1. 検知者（後から取り込んだ側）が `_共同開発/引き継ぎログ/YYYY-MM-DD_衝突_<file>.md` を作成
2. 両社の差分を記録
3. 小野寺社長 or 菅野社長にエスカレーションして判断仰ぎ
4. AI 単独で衝突解決しない（無断マージ禁止）

#### 🎯 最終決裁者ルール（受託契約の原則・経営判断）

**両社の本部長AI または両社長の判断が衝突した場合、最終決裁は菅野社長（発注者）**。
- 理由: AGO_SYSTEM_MANAGER は AGOグループ（菅野社長）から日本ネオン株式会社への受託案件であり、発注者の判断が優先される
- 例外: 法令違反・倫理違反・日本ネオン株式会社の信用毀損リスクがある場合は小野寺社長が拒否権を持つ
- 通常の仕様判断・優先順位・予算配分等は全て菅野社長判断優先

## 7. AI 同士の依頼方法

### 日本ネオン本部長AI から AGO本部長AI への依頼

直接通信はしない（AI 間は別 Claude セッションで隔離）。
代わりに：

1. 日本ネオン本部長AI が `_共同開発/引き継ぎログ/YYYY-MM-DD_日本ネオン→AGO_<件名>.md` を作成
2. SSD 経由（または git push）で AGO PC に反映
3. 小野寺社長が AGO 訪問時に AGO本部長AI に「`_共同開発/引き継ぎログ/` を確認してください」と伝える
4. AGO本部長AI が読んで対応

### AGO本部長AI から 日本ネオン本部長AI への依頼

逆方向も同様：

1. AGO本部長AI が `_共同開発/引き継ぎログ/YYYY-MM-DD_AGO→日本ネオン_<件名>.md` を作成
2. 小野寺社長が SSD 持ち帰り時に取り込む
3. 日本ネオン本部長AI（小野寺社長帰宅後）が読んで対応
4. 必要なら菅野社長から小野寺社長への伝達も活用

## 8. 緊急時の連絡経路

- **AI 単独で解決できない技術判断**: 担当社のAI が自社社長に判断仰ぎ
- **両社にまたがる経営判断**: 菅野社長 ⇄ 小野寺社長の直接対話
- **AI 暴走・データ消失等の緊急事態**: 双方の社長に即時報告

## 9. 完成度100%引き渡し時の本プロトコル取扱

- `_共同開発/` フォルダは AGO 側が維持・将来の追加開発用に活用
- 日本ネオン側は引き渡し後 6ヶ月の遠隔サポート期間を持つ（緊急時のみ・契約は別途）
